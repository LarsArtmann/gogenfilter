## A) FULLY DONE ✅

### Bug Fix: ScanProject WithFS

**File:** `scan.go:77`

`ScanProject` was not passing `WithFS(fsys)` to its internal `Filter`. This meant content-based detectors (ent, gqlgen, go-swagger, oapi-codegen, stringer, generic) silently failed when using `fstest.MapFS` or any custom `fs.FS`. Filename-based detectors (templ, protobuf) worked because they don't read content.

Root cause: The existing scan tests only tested filename-based generators, so the bug was invisible until this session added content-based generator coverage.

**Impact:** Any caller using `ScanProject` with a custom filesystem (e.g., `fstest.MapFS`) would have missed all content-only generators.

## G) Top #1 Question

**Is v3 in maintenance mode, or is there a v4 scope?**

The core library is functionally complete: 18 detectors, 99.8% coverage on existing code (98.3% total including new scan.go), comprehensive API surface (`Filter`, `FilterDetailed`, `FilterWithContent`, `DetectReason`, `DetectReasonFile`, `ScanProject`), and all known Go generators covered.

The three strategic options from ROADMAP.md are:
1. **Maintenance mode** — Only bug fixes, new detectors on request
2. **golangci-lint plugin** — The highest-impact integration play
3. **Custom detector registration API** — Extensibility for proprietary generators

This determines whether the next work is "close the CI gaps and declare done" or "build the next layer of the product."
