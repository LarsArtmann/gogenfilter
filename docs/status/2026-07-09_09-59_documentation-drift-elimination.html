05What We Should Improve

1. gendocs test coverage is not measured

cmd/gendocs now has 17 tests but its coverage is not included in the 98.4% number (that is library-only via go test -coverprofile=. .). Running go test -coverprofile=./cmd/gendocs/ would give us the gendocs coverage separately. The main() function itself is untested (it does file I/O on real project files), but all pure generation functions are tested.

2. detection.mdx table column widths are hardcoded

detectionFuncWidth = 26 and the header separator strings are hardcoded. If a future detector has a longer IsFuncName than IsCounterfeiterGenerated (24 chars + 2 backticks = 26), the table will misalign. The width should be computed dynamically from the data, or at minimum asserted in a test.

3. generators.mdx "Adding New Generators" section is stale

This section lists detector fields but omits the 4 new documentation fields and the websiteMetadata requirement. It should either be updated to match CONTRIBUTING.md or replaced with a link to CONTRIBUTING.md.

4. No test verifies generated output matches committed files

The CI docs job checks freshness via git diff --exit-code, but there is no local test that asserts "running gendocs produces output identical to what is committed." Such a test would catch drift before pushing. Could be a simple test that runs generateMDXTable(AllDetectorDocs()) and compares the result to the content between markers in the committed file.

5. gendocs writes files relative to CWD

gendocs uses hardcoded relative paths ("README.md", "website/src/data/generators.json", etc.). If run from a subdirectory, it will fail with "file not found." The //go:generate directive runs from the package directory (. for the root package), so this works in practice, but it is fragile. Using os.Getwd() or embed directives would be more robust.

6. Old status reports still reference deleted API pages

Historical status reports in docs/status/ and docs/planning/ contain references to api/filter, api/scan, etc. These are historical documents and should NOT be edited, but the website CI stale reference check scans --include='*.md' which could match these files if the search scope is broadened.

7. No gendocs marker validation test

There is no test that verifies all expected gendocs markers exist in all target files. If someone accidentally deletes a marker, gendocs will silently skip that section (the replaceSection function returns the original content unchanged when markers are missing). A test that checks for marker presence in all 5 files would catch this.

8. doc.go generator list line is very long

The generated line in doc.go is a single // comment containing all 18 generator names plus the generic suffix. It is 163 characters long. Go's gofmt does not break comment lines, and golines does not reformat comments. This is valid Go but harder to read in a terminal. Could be split across multiple // lines.
