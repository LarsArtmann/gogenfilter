8. Fixed Firebase deploy bash quote mangling

.github/workflows/website.yml:162-169 · commit c3d7784

The deploy step used inline expansion echo "${{ secrets.FIREBASE_SERVICE_ACCOUNT }}" which breaks when the JSON contains double quotes — bash closes the string early, producing malformed JSON. Switched to environment variable passing: 

# Before (broken):
echo "${{ secrets.FIREBASE_SERVICE_ACCOUNT }}" > "$RUNNER_TEMP/gcp-key.json"

# After (correct):
printf '%s' "$FIREBASE_SERVICE_ACCOUNT" > "$GOOGLE_APPLICATION_CREDENTIALS"
# with env: FIREBASE_SERVICE_ACCOUNT: ${{ secrets.FIREBASE_SERVICE_ACCOUNT }}

DTotally Fucked Up

Process mistakes and errors made during this session. Owning them.

Mistake

First commit lost the doc changes due to BuildFlow auto-fix

commit 65c13b6

I staged all files and committed, but the BuildFlow pre-commit hook ran oxfmt auto-fix which modified files after git add. The result: testdata/templ/page_templ.go was committed but the doc changes (README.md, detection.mdx, generators.mdx) were left unstaged. I had to make a second commit (4602483) for the docs. I should have verified the commit content with git show --stat HEAD before pushing.

Mistake

Used non-existent --body-file flag on first gh secret set

I guessed gh secret set --body-file "$KEYFILE" which doesn't exist in the installed gh version. The correct syntax is stdin: gh secret set NAME < "$KEYFILE". This generated a service account key that was immediately orphaned (had to delete it separately). I should have checked gh secret set --help first.

Mistake

Shredded temp key file before verifying secret was set

On the first failed attempt, I ran shred -u "$KEYFILE" at the end of a chained command — before checking whether gh secret set succeeded. It hadn't (wrong flag), so the key was generated in GCP but the local file was destroyed. Not a security issue (the key was deleted from GCP later), but sloppy ordering. The second attempt correctly sequenced: generate → set → verify → shred.

EWhat We Should Improve

Process Improvements

1
Always verify commit content before pushing Run git show --stat HEAD after committing, especially when pre-commit hooks run auto-fixes. This would have caught the lost doc changes immediately.
2
Check --help before guessing CLI flags 15 seconds of gh secret set --help would have saved a key-generation-and-deletion cycle.
3
Never chain shred/rm in a && sequence Destructive cleanup should only run after verifying the prior step succeeded.
4
Add a markdown render check to CI The gendocs freshness check validates that go generate output matches committed output — but not that the output renders correctly. A simple check for ^|| in README.md would catch the phantom-column bug.
5
Centralize GOEXPERIMENT=jsonv2 Currently duplicated across ci.yml and benchmark.yml. Consider a defaults: run: env: at the workflow level, or a reusable workflow, or migrate to Go 1.27 where json/v2 is native.
6
Use env: for secrets in all workflows The echo "${{ secrets.X }}" pattern is a known bash footgun. Audit all workflows and switch to env: + printf '%s'.

Codebase Observations

  • Service account key accumulation — every failed deploy attempt created a new key. No cleanup process exists. Consider a key rotation policy.
  • Lighthouse config is overly strict for current site state — downgrading to warnings is pragmatic but masks real issues. Track these as TODO items.
