## e) WHAT WE SHOULD IMPROVE

### Architecture Issues Found in Deep Audit

1. **`generatorExclusionReasons` map uses `string(FilterX)` keys** — `scan.go:192-207` maps generator names to human-readable reasons, but the keys are `string(FilterTempl)` etc. This is a separate parallel mapping that duplicates the relationship already encoded in the `detectors` table. If someone adds a detector but forgets this map, the exclusion reason silently falls back to "auto-generated code".

2. **`map[FilterOption]struct{}` as option set** — Used 22 times across source + tests. Go 1.26+ could use `maps` package or a typed `OptionSet` for clarity. The empty struct pattern is idiomatic but verbose when passed as parameters.

3. **`ExclusionPattern()` method is a local map literal** — `scan.go:37-53` creates a `map[FilterReason]string` on every call. Should be a package-level lookup table derived from the detectors table (like everything else).

4. **No `fmt.Stringer` on `FilterResult`** — Already has `String()` method. Good. But `GeneratedFile`, `Exclusion`, and `ScanResult` don't have `String()`.

5. **`ScanProject` silently swallows errors** — `scan.go:94` does `continue` on filter errors, making detection failures invisible to the caller. No error count, no warning.

6. **`detectReasonFromMap` is called even with empty content** — When `DetectReason` is called with empty content string, phase 1 runs (correct), then phase 2 calls `getContentBasedReasonWithTrace` with empty string, which always returns `ReasonNotFiltered`. The `needsContentCheck` guard only applies in the FS path, not the in-memory path.

7. **Test file count is high** — 23 test files (excluding testhelpers). `filter_test.go` alone is 733 lines. Could benefit from consolidation, but the existing split by concern (mapfs, content, edge, concurrent) is defensible.

8. **16 goconst warnings** — All accepted (cross-package constant sharing impossible with Go's test package split). No action needed but worth documenting the acceptance rationale.

9. **`gosec G304` on `readFile`** — False positive (library reads user-provided paths from `fs.FS`, not arbitrary filesystem access). Accepted but could add `#nosec` comment.

10. **`wrapcheck` on `fs.ReadFile` error** — Library wraps all errors but the raw `fs.ReadFile` return propagates unwrapped. Minor.

### Missing Features for Production Hardening

11. **No `FilterOption` → detector description mapping** — Users can enumerate options but can't get a human-readable description for UI display. The `generatorExclusionReasons` map is close but not exported.

12. **No `IsFiltered(reason FilterReason) bool` on `FilterResult`** — Minor ergonomic helper. `result.Reason == ReasonSQLC` works but `result.Is(ReasonSQLC)` reads better.

13. **`ScanResult` doesn't implement `fmt.Stringer`** — Would be useful for debugging/logging.

14. **No `FilteredCount()` or `ByGeneratorCount()` on `ScanResult`** — Callers must compute `len(result.Files)` themselves. Trivial but ergonomic.
