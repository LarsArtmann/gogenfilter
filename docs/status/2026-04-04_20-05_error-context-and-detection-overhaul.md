## A) FULLY DONE

### New Detector Support (types.go, detection.go)
- **FilterProtobuf** ‚Äî detects `.pb.go` and `_grpc.pb.go` files via filename + `// Code generated by` content
- **FilterMockgen** ‚Äî detects `_mock.go` / `mock_` prefix files + `// Code generated by MockGen` content
- **FilterStringer** ‚Äî detects `// Code generated by "stringer"` in content
- **FilterGeneric** ‚Äî fallback detector for any file with `// Code generated by` comment (per go.dev/s/generatedcode spec)
- All 4 integrated into `FilterAll` and the detection pipeline

### Table-Driven Detection Architecture (detection.go)
- Extracted `contentChecks` slice ‚Äî maps option ‚Üí reason ‚Üí check function, ordered specific‚Üígeneric
- Extracted `filenameChecks` slice ‚Äî maps option ‚Üí reason ‚Üí match function
- `getFilenameBasedReason()` and `getContentBasedReason()` now iterate tables instead of if-chains
- `needsContentCheck()` gates I/O ‚Äî skips file reads when only filename-based filters are enabled
- `allSpecificOptions` in types.go lists all tool-specific options for `FilterAll` expansion

### Typed Error Package (pkg/errors/errors.go)

## G) Top #1 Question I Cannot Figure Out Myself

**Is the `GetSQLOutputDirs` behavior change intentional as a breaking change?** The function previously logged parse failures via `slog.Warn` and continued, returning partial results. Now it returns an error immediately. This changes the API contract. Should we:
- (a) Keep the strict fail-fast behavior (current)
- (b) Revert to lenient log-and-continue
- (c) Provide both via `GetSQLOutputDirs` (strict) and `GetSQLOutputDirsLenient` (lenient)

This is a product decision that depends on downstream consumers.
