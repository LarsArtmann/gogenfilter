## d) TOTALLY FUCKED UP!

### 1. Website CI — md-go-validator Nix integration (CRITICAL)

**Status:** Every Website CI run fails. Every Dependabot PR is blocked.

**Root cause:** The Nix flake references `LarsArtmann/md-go-validator` (a private repo) via `fetchFromGitHub`. GitHub Actions cannot access private repo tarballs without authentication. The `nix run .#validate-docs` step fails with:

```
error: Failed to open archive (Source threw exception: error: unable to download
'https://api.github.com/repos/LarsArtmann/md-go-validator/tarball/4bead9193d414335d15e8406f80e289d513ed71f':
HTTP error 404)
```

**Why this is "totally fucked up":**
- The AGENTS.md says `continue-on-error: true` was the mitigation, but the Nix integration bypassed it
- The `flake.nix` `md-go-validator` input is hardcoded to a specific commit hash — if that commit is ever garbage-collected or the repo made private/archived, it breaks permanently
- The md-go-validator repo IS private (hence 404), but the Nix flake assumes public access
- This blocks ALL website deployments, ALL Dependabot merges, and ALL website CI validation

**Previous working state:** Before `d04b8c6` / `93ebc69` (Nix md-go-validator integration), the Website CI used `actions/checkout` with `PRIVATE_REPO_TOKEN` and `continue-on-error: true`. The Nix refactor broke this fallback.

## g) Top #1 Question I Cannot Figure Out Myself

### Why does the Nix flake's `fetchFromGitHub` for `md-go-validator` fail with 404 in CI, but the old `actions/checkout` with `PRIVATE_REPO_TOKEN` worked?

**What I know:**
- `LarsArtmann/md-go-validator` is a private GitHub repository
- The old Website CI used `actions/checkout` with `token: ${{ secrets.PRIVATE_REPO_TOKEN || github.token }}` and `continue-on-error: true`
- The new Nix integration uses `fetchFromGitHub` in `flake.nix` which calls the public GitHub tarball API (`https://api.github.com/repos/.../tarball/...`)
- Private repo tarballs require authentication; `fetchFromGitHub` supports `sha256` but not GitHub tokens

**What I don't know:**
- Is `md-go-validator` intended to remain private, or should it be public?
- Was the `continue-on-error: true` mitigation intentionally removed from the Nix step, or is it a bug?
- Is there a Nix-native way to authenticate `fetchFromGitHub` for private repos (e.g., `nix-prefetch-git` with SSH, or `builtins.fetchGit` with `GIT_SSH_COMMAND`)?
- Should the Nix approach be reverted to the old `actions/checkout` approach for CI, with Nix only used locally?

**This is blocking:**
- All Website CI runs
- All 7 open Dependabot PRs
- All future website deployments
- All doc validation in CI

**I need a decision from you on:**
1. Should `md-go-validator` be made public?

3. If keeping Nix, what's the authentication mechanism?
