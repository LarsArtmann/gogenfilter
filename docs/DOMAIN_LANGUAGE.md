# Domain Language — gogenfilter

A **Unified Language** for gogenfilter — shared across Consumer, Product Owner, Developer, and AI.
Inspired by Domain-Driven Design (DDD) Ubiquitous Language.

Every term below should mean the **same thing** to everyone who reads it.

## Glossary

| Term | Definition | Context |
|------|-----------|---------|
| gogenfilter | A Go library for detecting and filtering auto-generated code files | Project/product name |
| Generated file | A Go source file produced by a code generator (sqlc, protobuf, templ, etc.) | Core concept |
| Code generated comment | The standard `// Code generated by` comment per https://go.dev/s/generatedcode | Detection marker |
| Detector table | The `detectors` slice mapping each `FilterOption` to its `FilterReason`, filename matcher, and content checker | Internal architecture |
| Two-phase detection | Optimization: filename-based first (zero I/O), then content-based (reads file) | Performance |
| Generator | A specific tool that produces generated files (e.g., sqlc, templ, protobuf) | Enumeration |

## Entities

Objects with identity — types that carry state across operations.

| Term | Definition | Context |
|------|-----------|---------|
| Filter | A configured, immutable filter instance that orchestrates detection and pattern matching | Created via `NewFilter`, used per-file |
| FilterOption | A named type of generated code to detect (e.g., `FilterSQLC`, `FilterTempl`, `FilterAll`) | Configuration input |
| FilterReason | Why a file was or wasn't filtered (e.g., `ReasonSQLC`, `ReasonOutsideScope`, `ReasonNotFiltered`) | Detection output |
| FilterResult | The full outcome of filtering a file: filtered status, reason, path, and trace | Detailed output |
| ErrorCode | A specific error condition identifier (e.g., `CodeSQLCConfigParse`, `CodeProjectRootNotFound`) | Error handling |

## Value Objects

Immutable objects defined by their attributes.

| Term | Definition | Context |
|------|-----------|---------|
| FilterConfig | A functional option that configures a `Filter` during construction | Builder pattern |
| Detector | An entry in the detector table: option, reason, filename matcher, content checker | Table-driven |
| Include pattern | A glob pattern restricting scope — files not matching are filtered as `ReasonOutsideScope` | Scope restriction |
| Exclude pattern | A glob pattern that filters matching files regardless of detection | Override |
| Trace | A human-readable string explaining how detection reached its conclusion | Debugging |

## Commands

Actions the system performs.

| Term | Definition | Context |
|------|-----------|---------|
| Filter | Determine if a file should be excluded from analysis — returns `(bool, error)` | Primary API |
| FilterDetailed | Like Filter but returns `FilterResult` with reason and trace | Diagnostic API |
| FilterPaths | Batch filtering of multiple paths | Bulk API |
| DetectReason | Detect without I/O — caller provides content, returns `FilterReason` | Low-level API |
| FindSQLCConfigs | Discover sqlc.yaml/sqlc.yml files by walking directories | SQLC feature |
| GetSQLOutputDirs | Parse sqlc configs and extract output directories | SQLC feature |
| FindProjectRoot | Search parent directories for marker files (go.mod, etc.) | Infrastructure |

## Events

Outcomes that consumers act on.

| Term | Definition | Context |
|------|-----------|---------|
| Filtered | File was detected as generated (or matched an exclude/outside-scope pattern) | Positive result |
| Not filtered | File passed all detection checks — not generated | Negative result |

## Bounded Contexts

| Context | Description |
|---------|------------|
| **Detection** | The core engine: filename matching, content checking, the detector table. Answers "is this file generated, and by what?" |
| **Filtering** | The consumer-facing orchestrator: pattern matching (include/exclude), I/O (fs.FS), detection delegation. Answers "should this file be excluded?" |
| **SQLC Config** | Infrastructure: discovering, parsing, and extracting output directories from sqlc configuration files. Independent of detection engine. |
| **Error Reporting** | Branded errors with codes, sentinel errors for `errors.Is`, `ErrorCoder` interface. Cross-cutting concern used by all contexts. |
