Split-Brain Analysis

A first-principles audit of duplicated concepts, parallel type hierarchies, and divergent sources of truth across the gogenfilter data model — places where one concept is represented by two or more independent definitions that can silently drift out of sync.

00Summary

gogenfilter's data model is well-factored at the macro level — a single detectors table drives almost every enumeration in the library. But when you zoom in, the same conceptual entity is represented by two or more type-level definitions in at least six places. Each duplicate pair must be kept in lockstep by hand. None of the pairings are enforced by the type system, and only some are guarded by tests.

The most consequential split is also the most fundamental: every generator is described twice — once as a FilterOption (the configuration vocabulary) and once as a FilterReason (the outcome vocabulary). They share string values but live in separate type worlds, and the only thing linking them is a hand-maintained column in the detectors table. The other five are narrower but follow the same pattern: one concept, two definitions, zero compile-time coupling.

01What Is a Split-Brain?

A split-brain is any place where a single domain concept is encoded in two or more independent definitions, such that:

  1. The definitions can diverge without a compile error.
  2. Keeping them aligned is a human responsibility, not a type-system guarantee.
  3. A reader cannot tell which definition is authoritative.

Split-brains are not the same as duplication. Duplication is "the same code twice." A split-brain is "the same meaning encoded in two different vocabularies that are supposed to agree." That makes them subtler and more dangerous: tests pass until the day the two vocabularies disagree, and then they fail in places far from the edit.

02Findings

Each finding below lists the two (or more) definitions that purport to describe the same thing, the location of each, and the specific risk the split introduces.

SB1 — Dual Enum Types for One Concept

Critical

FilterOption and FilterReason are parallel string-typed enums

types.go:39-57, 120-154 · detection.go:17-22, 30-54

Every supported generator is named twice. Once as a FilterOption constant (the input vocabulary: "what to look for") and once as a FilterReason constant (the output vocabulary: "what was found"). Both are branded string types, both carry an identical set of 18 generator names, and the string values are deliberately identical (FilterSQLC = "sqlc" and ReasonSQLC = "sqlc").

The only thing connecting an option to its reason is the reason column in the detector struct, populated by hand for each of the 18 rows. The filterOptionToReason() function is a linear scan over that table. If a new detector is added with a mismatched pair (e.g. FilterNewGen = "newgen" but ReasonNewGen = "new-gen"), nothing breaks at compile time. The reason silently fails to round-trip.

Why it's a split-brain One concept ("sqlc-ness") has two authoritative spellings across two types. The detectors table is the de-facto join table, but Go gives it no privilege — either column can be edited independently.
Direction of fix Collapse into a single Generator identity type. An option and its reason become two views of the same value, not two values that are supposed to agree. See the unified model.

SB2 — Duplicate Directory-Skip Logic

Critical

Two near-identical "should I skip this directory?" functions with divergent rules

sqlc.go:183-190 (shouldSkipDirectory) · scan.go:188-194 (shouldSkipScanDir)

Both functions decide whether a directory should be skipped during a filesystem walk. They exist because two different walks (sqlc config discovery and scan-time file collection) each rolled their own. The logic is almost identical but not quite:

shouldSkipDirectory (sqlc.go:184)
shouldSkipScanDir (scan.go:188)
func shouldSkipDirectory(name string) bool {
  if name == "." {
    return false
  }
  return strings.HasPrefix(name, ".") ||
    name == nodeModulesDir || name == vendorDir
}
func shouldSkipScanDir(name string) bool {
  if strings.HasPrefix(name, ".") {
    return true
  }
  return name == vendorDir || name == nodeModulesDir
}

The sqlc version special-cases "." to return false before checking the hidden-directory prefix; the scan version does not, because its caller already guards path != "." before invoking it. The two functions are semantically equivalent today, but only because of an invariant enforced at the call site, not in the function itself. Add a new caller that invokes shouldSkipScanDir(".") directly and it will skip the root, silently returning zero files.

Why it's a split-brain "Which directories does a walk skip?" has two answers that must agree. They live in different files, were written by different authors, and have already drifted in structure even if not yet in behavior.

SB3 — Generator-Reason Map Keyed by Mixed Vocabularies

Important

generatorExclusionReasons() keys some entries by FilterOption, others by raw generator-name strings

scan.go:222-239 (generatorExclusionReasons)

The human-readable reason map uses string(FilterOption) keys for most generators, but two entries are keyed by raw detection-time constants instead: 

return map[string]string{
  string(FilterTempl):           templExclusionReason,
  // ... keyed by FilterOption value ...
  string(deepcopyGeneratorName): "deepcopy-gen generated code",  // raw constant!
  string(moqGeneratorName):      "moq generated mocks",          // raw constant!
  // ...
}

This happens to work because deepcopyGeneratorName = "deepcopy-gen" equals string(FilterDeepcopy), and likewise for moq. But that equality is an accident of string value, not a type-system fact. The map is keyed by string, erasing the distinction, and the lookup at scan.go:243,250,254 feeds in string(result.Reason). A future generator whose option value differs from its detection marker would silently miss the lookup and fall through to the generic "auto-generated code" default.

Why it's a split-brain The authoritative key for "what is this generator called?" is sometimes the FilterOption vocabulary, sometimes the detection.go marker constant vocabulary. The string map hides which is which.

SB4 — Sentinel Errors Duplicated as Code Constants

Important

Every error condition is named twice: once as an ErrorCode constant, once as a sentinel error variable

errors.go:34-43 (codes) · errors.go:54-63 (sentinels)

Each error class has a paired definition: CodeSQLCConfigParse (the enum) and ErrSQLCConfigParse (the sentinel). The sentinel is a *SQLCConfigError with only the Code field set; matching is by code equality via the Is() method. The two must stay paired: 

// errors.go:39 + 59
CodeSQLCConfigParse ErrorCode = "sqlc_config_parse"
ErrSQLCConfigParse         = &SQLCConfigError{Code: CodeSQLCConfigParse}

This is a deliberate design (the sentinels enable errors.Is without allocating), and the Is() methods compare codes rather than pointer identity, so it is safer than it looks. But the pairing is still hand-maintained: adding CodeFoo without ErrFoo (or vice versa) compiles cleanly and produces a class of errors that cannot be matched with errors.Is.

Why it's a split-brain "What kinds of things can go wrong?" has two registries: the code enum and the sentinel var-block. They must enumerate the same set. Nothing checks that they do.
Mitigating factor The errors_sentinel_test.go and errors_unwrap_test.go suites verify the round-trip for every pair. This is the one split-brain that is test-guarded, which is why it is rated Important rather than Critical.

SB5 — ExclusionPattern Lives on FilterReason, Not FilterOption

Important

ExclusionPattern() is a method on FilterReason, but the parallel lookup table is conceptually a property of the generator

scan.go:56-82 (exclusionPatterns map, ExclusionPattern method)

The regex used to derive a golangci-lint exclusion is keyed by FilterReason. But it is read in scan.go:242 via FilterReason(generator).ExclusionPattern() where generator is a raw string that was produced by string(result.Reason) at scan.go:130. The chain works, but it threads the generator identity through string three times: reason → string → FilterReason → method receiver.

Meanwhile, generatorExclusionReasons() (SB3) is keyed by what is effectively FilterOption. So the two exclusion-related lookups in the same file disagree about which type is the key. A reader trying to answer "what do I key exclusion data by?" gets two answers depending on which function they're looking at.

Why it's a split-brain "Generator-scoped metadata" (exclusion pattern, human-readable reason) is attached to FilterReason in one place and to FilterOption in another, within the same file, for the same conceptual purpose.

SB6 — Two String Vocabularies for the Same Generator Names

Nice to have

detection.go marker constants restate generator names that FilterOption already names

detection.go:115-140 (marker constants) · types.go:59-118 (FilterOption constants)

deepcopyGeneratorName = "deepcopy-gen", moqGeneratorName = "moq", wireGeneratorName = "Wire", mockgenGeneratorName = "MockGen", etc. are used to match // Code generated by X comments. Several of these differ in case or specificity from their FilterOption siblings ("Wire" vs FilterWire = "wire"; "MockGen" vs FilterMockgen = "mockgen").

This is not strictly wrong — the detection marker must match the literal string the generator emits, which is a different concern from the option's stable identifier. But it does mean there are two string vocabularies for "what is this generator called," and the mapping between them is implicit. The naming suggests they should agree (both use the generator name), which invites the assumption that they do.

Why it's a split-brain (mild) The two vocabularies describe genuinely different things (comment literal vs. public identifier), but their visual similarity masks the distinction. A contributor renaming FilterWire will not realize wireGeneratorName is unrelated.

04Toward a Unified Model

The dominant root cause is that a "generator" is represented as two branded string types that are supposed to agree. The clean fix is to make the generator identity a single value from which both the option and the reason are derived views — not two values joined by a hand-maintained table.

Today: two types, one join table
Proposal: one identity, two views
// types.go
type FilterOption string
type FilterReason string

const (
  FilterSQLC FilterOption = "sqlc"
  ReasonSQLC FilterReason = "sqlc"
  // ...18 generators, named twice...
)

// detection.go: hand-maintained join
type detector struct {
  option FilterOption
  reason FilterReason
  ...
}
// Proposed: one identity type
type Generator string

const GeneratorSQLC Generator = "sqlc"

// FilterOption and FilterReason become
// type aliases / views over Generator,
// preserving the public API.
type FilterOption = Generator
type FilterReason = Generator

// detector now carries one identity,
// not two that must agree.
type detector struct {
  gen Generator
  ...
}

The type-alias approach is the lowest-risk migration: existing FilterSQLC / ReasonSQLC references keep compiling because the aliases resolve to the same underlying type, but the source of truth collapses to one constant per generator. The detectors table loses its reason column entirely; filterOptionToReason becomes the identity function.

For SB2, the fix is mechanical: delete one function, point both callers at the other, and add the "." guard inside the survivor so the invariant does not depend on the call site. For SB4, consider a code-generation step or a test (already largely present) that asserts every Code* has a paired Err* and vice versa.

Strengths Worth Preserving

The review also surfaced things the model gets right. Any remediation should preserve them:

+
Single detectors table as the central registry 18 generators defined in one place, with derived enumerators (AllFilterOptions, AllGeneratorOptions, AllFilterReasons). This is the right architecture; the split-brain is that the table has two columns that should be one.
+
Branded string types, not raw strings FilterOption, FilterReason, ErrorCode, SQLCOperation are all distinct named types. This prevents the vast majority of accidental cross-assignment. The split-brains survive despite this discipline, which is a testament to how much it already catches.
+
errors.Is support via code comparison The Is() methods compare .Code fields rather than pointer identity, so sentinel matching is robust to allocation. The sentinel/code pairing (SB4) is the most defensible of the split-brains.
+
Immutable, package-level lookup tables detectors, exclusionPatterns, and the marker constants are all package-level var/const blocks guarded by nolint:gochecknoglobals. No mutable state, no initialization order surprises.

05Remediation Plan

Ordered by impact-to-risk ratio. Steps 1 and 2 are low-risk, high-clarity wins that should be done first. Step 3 is the architectural fix that dissolves the cluster; it is a larger change but remains API-compatible via type aliases.

1

Unify the directory-skip functions (SB2)

Delete shouldSkipScanDir, fold its "." guard into shouldSkipDirectory, and point the scan.go caller at the survivor. Add a unit test that calls the function directly with ".", ".hidden", "vendor", "node_modules", and "src". Lowest-risk, highest immediate clarity gain.

2

Re-key generatorExclusionReasons by FilterReason (SB3)

Replace the two raw-constant keys (deepcopyGeneratorName, moqGeneratorName) with string(FilterDeepcopy) and string(FilterMoq) so every entry uses the same vocabulary. Add a test that asserts every key in the map is a member of AllFilterReasons(). This is a one-line semantic fix plus a guard test.

3

Introduce a Generator identity type (SB1, SB5, SB6)

Add type Generator string with one constant per generator. Make FilterOption and FilterReason type aliases for Generator (or keep them distinct named types with a single shared constant pool). Drop the reason column from detector. Move ExclusionPattern() and the human-readable reason onto Generator so generator-scoped metadata has one home. API-compatible via aliases; dissolves SB1, SB3, SB5, and the vocabulary confusion of SB6 in one stroke.

4

Add a sentinel/code pairing invariant test (SB4)

The existing errors_sentinel_test.go covers round-tripping. Add a reflection-based test that enumerates every Code* constant and asserts a corresponding Err* sentinel exists (and vice versa), so the pairing cannot silently drift when a new error class is added.

5

Document the marker-vs-identifier distinction (SB6)

If the detection-marker constants must remain distinct from option values (they often must, due to case differences in emitted comments), add a doc comment block to detection.go explicitly stating that *GeneratorName constants are comment-literal matchers, not public identifiers, and that they are intentionally allowed to differ from FilterOption values. Naming alone will not convey this.

Bottom line

gogenfilter's data model is closer to right than wrong. The single detectors table is a strong spine. But six places encode one concept twice, and four of those trace to a single decision — representing generator identity as two branded string types. Collapsing that decision via a Generator identity type (with FilterOption / FilterReason as aliases) is the highest-leverage change available, and it is doable without breaking the public API.
